ផែនការធានាគុណភាព និងការធ្វើតេស្ត (Quality Assurance / Test Plan)

ប្រព័ន្ធគ្រប់គ្រងឥណទាន និងគណនេយ្យ

១. សេចក្តីផ្តើម

ឯកសារនេះរៀបរាប់ពីយុទ្ធសាស្ត្រ និងវិធីសាស្រ្តដែលនឹងត្រូវប្រើប្រាស់ដើម្បីធានាគុណភាពនៃប្រព័ន្ធគ្រប់គ្រងឥណទាន និងគណនេយ្យ។ គោលបំណងគឺដើម្បីកំណត់អត្តសញ្ញាណ កត់ត្រា និងដោះស្រាយកំហុសឆ្គង (Bugs) មុនពេលប្រព័ន្ធត្រូវបានដាក់ឱ្យដំណើរការ។

២. គោលបំណងនៃការធ្វើតេស្ត

  • ធានាថាមុខងារទាំងអស់ដំណើរការត្រឹមត្រូវតាមតម្រូវការដែលបានកំណត់។
  • ធានាថាប្រព័ន្ធមានស្ថិរភាព និងប្រសិទ្ធភាពក្នុងការឆ្លើយតប។
  • កំណត់អត្តសញ្ញាណ និងកាត់បន្ថយកំហុសឆ្គង។
  • ធានាថាប្រព័ន្ធមានសុវត្ថិភាពពីការគំរាមកំហែងទូទៅ។
  • ផ្ដល់ទំនុកចិត្តដល់អតិថិជនអំពីគុណភាពនៃផលិតផល។

៣. ប្រភេទនៃការធ្វើតេស្ត (Types of Testing)

  • ៣.១. Unit Testing

    ធ្វើតេស្តលើផ្នែកតូចៗនីមួយៗនៃកូដ (ឧទាហរណ៍៖ Functions, Methods) ដើម្បីធានាថាពួកវាដំណើរការត្រឹមត្រូវដោយឯករាជ្យ។

  • ៣.២. Integration Testing

    ធ្វើតេស្តលើការភ្ជាប់គ្នារវាងម៉ូឌុលផ្សេងៗនៃប្រព័ន្ធ (ឧទាហរណ៍៖ Backend API ជាមួយ Database, Frontend ជាមួយ Backend API) ដើម្បីធានាថាពួកវាធ្វើការជាមួយគ្នាបានល្អ។

  • ៣.៣. Functional Testing

    ធ្វើតេស្តថាតើមុខងារនីមួយៗរបស់ប្រព័ន្ធ (ឧទាហរណ៍៖ ការបង្កើតកម្ចីថ្មី ការកត់ត្រាការទូទាត់) ដំណើរការបានត្រឹមត្រូវតាមតម្រូវការរបស់អតិថិជន។

  • ៣.៤. Usability Testing

    វាយតម្លៃភាពងាយស្រួលប្រើប្រាស់នៃចំណុចប្រទាក់អ្នកប្រើប្រាស់ (UI/UX) របស់ប្រព័ន្ធ។

  • ៣.៥. Performance Testing

    វាយតម្លៃល្បឿន និងភាពឆ្លើយតបរបស់ប្រព័ន្ធនៅក្រោមបន្ទុកការងារផ្សេងៗ (ឧទាហរណ៍៖ ការបញ្ចូលទិន្នន័យច្រើន ការចូលប្រើប្រាស់ដំណាលគ្នា)។

  • ៣.៦. Security Testing

    ធ្វើតេស្តដើម្បីកំណត់អត្តសញ្ញាណភាពងាយរងគ្រោះផ្នែកសុវត្ថិភាព (ឧទាហរណ៍៖ SQL Injection, Cross-Site Scripting, Unauthorized Access)។

  • ៣.៧. User Acceptance Testing (UAT)

    អតិថិជននឹងធ្វើតេស្តប្រព័ន្ធដោយខ្លួនឯងដើម្បីធានាថាវាបំពេញតាមតម្រូវការអាជីវកម្មរបស់ពួកគេ។

៤. ដំណើរការធ្វើតេស្ត (Testing Process)

  1. ៤.១. ការរៀបចំ Test Cases

    បង្កើត Test Cases លម្អិតសម្រាប់មុខងារនីមួយៗ។ Test Case នីមួយៗនឹងរួមបញ្ចូល:

    • លេខរៀង Test Case
    • ការពិពណ៌នា
    • ជំហានក្នុងការធ្វើតេស្ត
    • លទ្ធផលរំពឹងទុក
    • លទ្ធផលជាក់ស្តែង (ពេលធ្វើតេស្ត)
    • ស្ថានភាព (Pass/Fail)
    • កំណត់ចំណាំ/លេខ Bug ID

  2. ៤.២. ការអនុវត្តការធ្វើតេស្ត (Test Execution)

    អនុវត្ត Test Cases ទាំងអស់នៅលើបរិយាកាសតេស្ត (Staging Environment)។

  3. ៤.៣. ការកត់ត្រា Bug (Bug Reporting)

    រាល់ Bug ដែលរកឃើញនឹងត្រូវបានកត់ត្រាក្នុងប្រព័ន្ធគ្រប់គ្រង Bug (ឧទាហរណ៍៖ Jira, Trello, ឬ Spreadsheet) រួមមាន៖ ការពិពណ៌នា លេខជំហានដើម្បីបង្កើត Bug ឡើងវិញ រូបភាព/វីដេអូ និងកម្រិតធ្ងន់ធ្ងរ។

  4. ៤.៤. ការដោះស្រាយ Bug និងការធ្វើតេស្តឡើងវិញ (Bug Resolution & Retesting)

    អ្នកអភិវឌ្ឍន៍នឹងកែសម្រួល Bug ហើយក្រុម QA នឹងធ្វើតេស្តឡើងវិញដើម្បីធានាថា Bug ត្រូវបានដោះស្រាយ និងមិនមាន Bug ថ្មីកើតឡើង។

៥. បរិយាកាសធ្វើតេស្ត (Test Environment)

ការធ្វើតេស្តនឹងត្រូវបានអនុវត្តនៅលើបរិយាកាសដូចទៅនឹងបរិយាកាស Production តាមដែលអាចធ្វើទៅបាន ដើម្បីធានាភាពស៊ីសង្វាក់គ្នា។

  • Operating System: [ឧទាហរណ៍៖ Windows Server 2022, Ubuntu 22.04 LTS]
  • Web Browser: [ឧទាហរណ៍៖ Chrome (កំណែចុងក្រោយ), Firefox (កំណែចុងក្រោយ), Edge (កំណែចុងក្រោយ)]
  • Database: [ឧទាហរណ៍៖ PostgreSQL 14]
  • Network Configuration: ត្រូវតែមានល្បឿនអ៊ីនធឺណិតគ្រប់គ្រាន់។

៦. ការទទួលខុសត្រូវ (Responsibilities)

  • អ្នកអភិវឌ្ឍន៍: ទទួលខុសត្រូវលើ Unit Testing និងការដោះស្រាយ Bug ។
  • អ្នក/ក្រុម QA: ទទួលខុសត្រូវលើការរៀបចំ Test Plan, ការអនុវត្ត Integration/Functional/Performance/Security Testing និងការកត់ត្រា Bug ។
  • អតិថិជន: ទទួលខុសត្រូវលើ User Acceptance Testing (UAT)។

៧. សេចក្តីសន្និដ្ឋាន

ផែនការធានាគុណភាពនេះ នឹងជួយធានាថាប្រព័ន្ធគ្រប់គ្រងឥណទាន និងគណនេយ្យដែលបានអភិវឌ្ឍន៍ នឹងមានគុណភាពខ្ពស់ អាចទុកចិត្តបាន និងបំពេញតាមតម្រូវការរបស់ [ឈ្មោះក្រុមហ៊ុនអតិថិជន]។
